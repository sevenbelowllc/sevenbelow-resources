**Release Radar** is a lightweight internal tool that summarizes GitHub PR activity, triages issues, and drafts weekly stakeholder update emails. It is built on the Claude Code ecosystem — skills, hooks, and the Claude API — and serves as the single codebase for two homework submissions in the DataExpert Claude Code bootcamp.

## 2. Architecture

### Dual-Mode Operation

Release Radar operates in two modes:

1. **Interactive mode** — User invokes skills directly in Claude Code (e.g., `/triage-issue`). Hooks fire guardrails automatically before and after skill execution.
2. **Batch mode** — `python release_radar.py --weekly` calls the Claude API and GitHub API programmatically, applies guardrails via Python classes, and outputs a complete weekly report.

### Data Flow

```
GitHub REST API / Mock JSON fixtures
       |
       v
  gh_adapter.py (normalize to common schema)
       |
       v
  [PRE-GUARDRAILS: insufficient_context, pii_redaction on input]
       |
       v
  Skill invocation (interactive) / Claude API call (batch)
       |
       v
  [POST-GUARDRAILS: schema_validation, citation, uncertainty, pii_redaction on output]
       |
       v
  Structured output / stakeholder email
```

### Technology Stack

- **Python 3.11+** — core runtime
- **Claude Code** — interactive skill execution, hooks
- **Anthropic SDK (`anthropic`)** — programmatic Claude API calls in batch mode
- **GitHub REST API** — live data ingestion via `GITHUB_TOKEN`
- **`python-dotenv`** — environment variable management
- **`requests`** — HTTP client for GitHub API
- **`jsonschema`** — schema validation guardrail

### API Keys

Stored in `.env` (gitignored), template in `.env.example`:

```
ANTHROPIC_API_KEY=sk-ant-...
GITHUB_TOKEN=ghp_...
```

## 4. Skills Design

### 4.1 triage-issue

**Purpose:** Classify a GitHub issue by severity, priority, suggest labels and owner.

**Input:**
```json
{
  "title": "Login fails on Safari",
  "body": "Users report 500 errors when...",
  "comments": ["I can reproduce on...", "Same here on iOS..."],
  "labels_existing": ["bug", "frontend", "backend", "urgent", "low-priority"]
}
```

**Output Schema:**
```json
{
  "status": "ok",
  "severity": "high|medium|low",
  "priority": "p0|p1|p2|p3",
  "labels": ["bug", "frontend"],
  "recommended_owner": "frontend-team",
  "confidence": "high|medium|low",
  "uncertainty_flags": ["limited reproduction data"],
  "citations": [{"claim": "affects Safari", "source": "issue body: 'Users report 500 errors'"}],
  "reasoning": "..."
}
```

**Insufficient context fallback:**
```json
{
  "status": "insufficient_context",
  "missing": ["body", "reproduction steps"],
  "suggestion": "Please provide a detailed description and steps to reproduce."
}
```

### 4.2 summarize-pr

**Purpose:** Generate a concise technical summary and risk checklist from PR data.

**Input:**
```json
{
  "title": "Migrate auth to OAuth2",
  "description": "Replaces session-based auth with...",
  "diff_snippets": ["- session.create()\n+ oauth.authorize()", "..."],
  "files_changed": ["auth/provider.py", "tests/test_auth.py"],
  "additions": 245,
  "deletions": 189
}
```

**Output Schema:**
```json
{
  "status": "ok",
  "summary": "Replaces session-based authentication with OAuth2...",
  "risk_checklist": [
    {"risk": "Breaking change for existing sessions", "severity": "high", "mitigation": "Migration script included"},
    {"risk": "New dependency on oauth-lib", "severity": "low", "mitigation": "Pinned to v2.1"}
  ],
  "confidence": "high|medium|low",
  "citations": [{"claim": "migration script included", "source": "diff: auth/migrate.py added"}]
}
```

### 4.3 digest-commits

**Purpose:** Organize a list of commits into a structured digest with sections.

**Input:**
```json
{
  "commits": [
    {"sha": "abc1234", "message": "fix: login timeout on Safari", "author": "alice", "date": "2026-04-07"},
    {"sha": "def5678", "message": "feat: add OAuth2 provider", "author": "bob", "date": "2026-04-08"}
  ],
  "date_range": {"start": "2026-04-05", "end": "2026-04-11"}
}
```

**Output Schema:**
```json
{
  "status": "ok",
  "sections": {
    "what_changed": "2 commits: 1 bug fix (login timeout), 1 new feature (OAuth2)...",
    "risk_impact": "Auth system changes affect all users. Safari fix is low-risk...",
    "action_needed": "QA should verify OAuth2 flow before release..."
  },
  "confidence": "high|medium|low",
  "citations": [
    {"claim": "login timeout fixed", "source": "commit abc1234: 'fix: login timeout on Safari'"}
  ]
}
```

### 4.4 draft-email

**Purpose:** Produce a polished stakeholder email from digest and PR summary outputs.

**Input:**
```json
{
  "digest": { "...output from digest-commits..." },
  "pr_summaries": [ "...outputs from summarize-pr..." ],
  "recipient_context": "engineering leadership, non-technical stakeholders",
  "week": "2026-04-05 to 2026-04-11"
}
```

**Output Schema:**
```json
{
  "status": "ok",
  "subject": "Engineering Update: Week of April 5-11, 2026",
  "body": "Hi team,\n\nHere's what the engineering team shipped this week...",
  "pii_redacted": true,
  "confidence": "high|medium|low",
  "citations": [{"claim": "...", "source": "..."}]
}
```

### 4.5 handoff

**Purpose:** Capture current project state for session/task handoff. Already implemented — moves from `claude-code-homework/handoff/SKILL.md` into `claude_skills/handoff/SKILL.md`.

**Output:** Structured markdown written to `memory/HANDOFF.md`.

## 5. Guardrails Design

### 5.1 Shared Interface

```python
class Guardrail:
    """Base class for all guardrails."""
    
    def check_input(self, data: dict) -> dict:
        """Pre-processing check. Returns modified data or raises GuardrailError."""
        return data
    
    def check_output(self, data: dict) -> dict:
        """Post-processing check. Returns modified data or raises GuardrailError."""
        return data
```

### 5.2 Execution Order

**Pre-processing (before LLM call):**
1. `insufficient_context` — reject inputs that lack required fields
2. `pii_redaction` — scrub PII from input before sending to Claude

**Post-processing (after LLM call):**
3. `schema_validation` — validate output matches expected JSON schema
4. `citation` — verify every claim references source data
5. `uncertainty` — validate confidence field consistency
6. `pii_redaction` — scrub any PII that leaked into output

### 5.3 Guardrail Details

#### PII Redaction (`pii_redaction.py`)
- **Regex pass — structured patterns:**
  - Email addresses (`\b\w+@\w+\.\w+\b`)
  - Phone numbers (US and international formats)
  - Social Security Numbers (`XXX-XX-XXXX`)
  - Credit card numbers (16-digit patterns, Luhn-validated)
  - IP addresses (IPv4 and IPv6)
  - API keys and tokens: `sk-ant-`, `ghp_`, `github_pat_`, `xoxb-`, `xoxp-`, `xoxs-`, `AKIA` (AWS)
  - Private keys / certificates (`BEGIN RSA`, `BEGIN SSH`, `BEGIN CERTIFICATE`)
  - Database connection strings (`postgres://`, `mysql://`, `mongodb://`, `redis://`)
  - Internal URLs / hostnames (`*.internal`, `*.corp`, `*.local`, `*.lan`)
  - Physical / mailing addresses (street number + name patterns)
- **Claude API pass:** send text to Claude with instruction to identify contextual PII (names in sensitive context, internal project names, etc.)
- **Action:** replace with typed tags: `[REDACTED-EMAIL]`, `[REDACTED-PHONE]`, `[REDACTED-SSN]`, `[REDACTED-KEY]`, `[REDACTED-NAME]`, `[REDACTED-URL]`, `[REDACTED-CONNECTION-STRING]`, etc.
- Runs on both input and output

#### Schema Validation (`schema_validation.py`)
- Per-skill JSON schemas defined in Python dicts
- Uses `jsonschema` library to validate
- On failure: returns structured error with which fields are invalid/missing
- Runs on output only

#### Uncertainty Statements (`uncertainty.py`)
- Validates `confidence` field exists in output
- Cross-checks: if output contains hedging language ("might", "possibly", "unclear") but confidence is "high", flags inconsistency
- Validates `uncertainty_flags` array is present and non-empty when confidence is not "high"
- Runs on output only

#### Citation to Source (`citation.py`)
- Validates `citations` array exists and is non-empty
- Each citation must have `claim` and `source` fields
- Validates that `source` references something present in the input data (SHA, PR number, quote)
- Runs on output only

#### Insufficient Context Fallback (`insufficient_context.py`)
- Per-skill minimum input requirements (e.g., triage-issue needs at least `title`)
- If requirements not met: short-circuits with `{status: "insufficient_context", missing: [...], suggestion: "..."}`
- Does not call the LLM at all when triggered
- Runs on input only

### 5.4 Hook Integration

Hooks registered in `.claude/settings.json`:

```json
{
  "hooks": {
    "pre_tool_call": [
      {
        "pattern": "claude_skills/*",
        "command": "python guardrails/hooks/pre_tool_guardrail.py"
      }
    ],
    "post_tool_call": [
      {
        "pattern": "claude_skills/*",
        "command": "python guardrails/hooks/post_tool_guardrail.py"
      }
    ]
  }
}
```

> **Note:** The exact hook registration syntax must be verified against Claude Code's current hook API at implementation time. The pattern matching and event names above are illustrative — the implementation plan should confirm the correct format.

Hook scripts are thin wrappers that:
1. Read tool input/output from stdin
2. Import and run the appropriate guardrail classes
3. Return modified data or error to stdout

## 6. Data Layer

### 6.1 Mock Data

All mock data lives in `data/mock/` as JSON files.

#### `issues.json` — 15-25 GitHub Issues
- Mix of severities (critical, high, medium, low)
- Mix of completeness (some with full body + comments, some sparse/title-only)
- At least 2 issues with embedded PII (author emails in body, API keys in code blocks)
- At least 3 issues intentionally sparse to trigger insufficient-context guardrail

#### `pull_requests.json` — 15-25 Pull Requests
- Mix of sizes (small refactors to large features)
- Include diff snippets (abbreviated, not full diffs)
- At least 2 PRs with PII in diff (hardcoded emails, tokens)
- Mix of risk levels (breaking changes, dependency updates, cosmetic fixes)

#### `commits.json` — 50-75 Commits
- Spanning a simulated week
- Multiple authors (5-8 simulated developers)
- Mix of commit types: feat, fix, chore, docs, refactor, test
- Conventional commit format where possible
- Realistic distribution (more fixes than features, occasional chore)

### 6.2 Live Data Adapter

`gh_adapter.py` provides a unified interface:

```python
class GitHubAdapter:
    def __init__(self, token: str = None, mock: bool = False):
        """If mock=True, loads from data/mock/. Otherwise uses GitHub API."""
    
    def get_issues(self, repo: str, since: str) -> list[dict]:
        ...
    
    def get_pull_requests(self, repo: str, state: str = "merged", since: str = None) -> list[dict]:
        ...
    
    def get_commits(self, repo: str, since: str) -> list[dict]:
        ...
```

Output schema is identical regardless of source (mock or live). Skills and guardrails don't know the difference.

## 7. CLI Orchestrator

`scripts/release_radar.py` — the batch-mode entry point.

### Commands

```bash
# Full weekly report (default: last 7 days, mock data)
python scripts/release_radar.py --weekly

# Full weekly report with live data
python scripts/release_radar.py --weekly --repo owner/repo --live

# Individual operations
python scripts/release_radar.py triage --input data/mock/issues.json
python scripts/release_radar.py summarize --input data/mock/pull_requests.json
python scripts/release_radar.py digest --input data/mock/commits.json
python scripts/release_radar.py email --input data/mock/commits.json --prs data/mock/pull_requests.json

# Use specific date range
python scripts/release_radar.py --weekly --since 2026-04-01 --until 2026-04-11
```

### Weekly Report Flow

1. Load data (mock or live via `gh_adapter.py`)
2. Run `triage-issue` on all open issues → collect results
3. Run `summarize-pr` on all merged PRs → collect results
4. Run `digest-commits` on all commits → collect digest
5. Run `draft-email` with digest + PR summaries → produce email
6. Apply all guardrails at each step
7. Output final report to stdout or file

### 8.2 Unit Tests

Unit tests cover all guardrails and the data adapter without making API calls.

#### PII Redaction Tests (`test_pii_redaction.py`)
- Each of the 12+ pattern categories has at least one positive and one negative test
- Verifies correct replacement tags (`[REDACTED-EMAIL]`, `[REDACTED-KEY]`, etc.)
- Tests that non-PII content passes through unchanged
- Tests mixed content (PII embedded in normal text)
- Tests edge cases: partial matches, lookalikes that should NOT be redacted

#### Schema Validation Tests (`test_schema_validation.py`)
- Valid output for each skill passes validation
- Missing required fields are caught
- Wrong field types are caught
- Extra fields are allowed (forward compatibility)
- Tests the `insufficient_context` response schema separately

#### Uncertainty Tests (`test_uncertainty.py`)
- High confidence with no hedging language passes
- High confidence with hedging language ("might", "possibly") is flagged
- Low/medium confidence with empty `uncertainty_flags` is flagged
- Validates `confidence` field is one of `high|medium|low`

#### Citation Tests (`test_citation.py`)
- Output with valid citations (referencing input SHAs, PR numbers) passes
- Output with empty citations array fails
- Citations referencing data NOT in input are flagged
- Each citation has both `claim` and `source` fields

#### Insufficient Context Tests (`test_insufficient_context.py`)
- Issue with title only → triggers fallback
- Issue with title + body → passes
- PR with no diff snippets → triggers fallback
- Commits with empty list → triggers fallback
- Verifies `missing` field lists exactly what's absent

#### GitHub Adapter Tests (`test_gh_adapter.py`)
- Mock mode loads from `data/mock/` correctly
- Output schema matches between mock and live modes (structure only, mock test)
- Missing mock files raise clear errors
- Date filtering works correctly

#### Guardrail Chain Tests (`test_guardrail_chain.py`)
- Pre-guardrails execute in order: insufficient_context → pii_redaction
- Post-guardrails execute in order: schema_validation → citation → uncertainty → pii_redaction
- A failing pre-guardrail short-circuits (no LLM call)
- A failing post-guardrail returns structured error, not raw output

### 8.3 Integration Tests (CLI)

Integration tests invoke the CLI script as a subprocess and verify end-to-end behavior against mock data. These tests call the Claude API.

#### CLI Tests
- `test_cli_triage.py` — runs `python scripts/release_radar.py triage --input data/mock/issues.json`, verifies JSON output matches schema, guardrails applied
- `test_cli_summarize.py` — runs `summarize` command, verifies PR summary output
- `test_cli_digest.py` — runs `digest` command, verifies commit digest sections
- `test_cli_email.py` — runs `email` command, verifies stakeholder email output with PII redacted
- `test_cli_weekly.py` — runs `--weekly` full pipeline, verifies complete report
- `test_guardrail_hooks.py` — verifies hook scripts accept stdin, run guardrails, return modified data

#### CLI Test Expectations
- Exit code 0 on success
- Exit code 1 on guardrail failure with structured error to stderr
- JSON output parseable and schema-valid
- PII patterns absent from all output
- Citations present in all output

### 8.4 Linting

```bash

# Formatter

# Type checking
```

```bash

# Integration tests (requires .env with API keys)

# All tests
```

### 8.6 Validation Gate

**No implementation step is complete until:**
1. All existing unit tests pass
2. New code has corresponding unit tests
3. `ruff check .` and `ruff format --check .` pass
4. Integration tests pass for any modified CLI commands
5. No regressions in previously passing tests

This gate is enforced in the CLAUDE.md Validation section — any AI agent working on this project must run validation before claiming a task is done.

## 12. Risks and Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Claude API rate limits during batch mode | Batch run fails mid-way | Implement retry with backoff; process items sequentially not in parallel |
| Hook execution adds latency to interactive skills | Slow UX | Guardrail hooks should complete in <2s; cache regex patterns |
| Mock data doesn't trigger all guardrail paths | Incomplete demo | Intentionally craft mock data with PII, sparse issues, hedging language |
| Schema drift between skills and guardrails | Validation false positives | Single source of truth for schemas in `guardrails/schemas/` |
| GitHub API auth issues | Live mode fails | Fall back to mock data gracefully; clear error messages |

## 13. Out of Scope

- Web UI (CLI only)
- Persistent storage / database
- Multi-user support
- Scheduling / cron (manual invocation only)
- Slack / Teams integration
- Deployment to cloud infrastructure
