# Release Radar — Documentation Rationale (HW2)

## Overview

This project produces two primary documents: `CLAUDE.md` (an AI agent control file at the
project root) and `memory/HANDOFF.md` (a human-readable session handoff generated by the
`handoff` skill). They serve different audiences, are optimized for different reading patterns,
and are maintained by different mechanisms.

---

## Why Two Documents?

### CLAUDE.md is written for AI agents

When a new Claude Code session opens in this project, Claude reads `CLAUDE.md` before
responding to any user request. This means CLAUDE.md must be optimized for machine
comprehension: it should be prescriptive, unambiguous, and dense with behavioral instructions.

The questions it must answer are:
- What is this project? (Project Overview)
- What commands must I run to validate my work? (Validation section)
- Where does the code live? (Folder Map)
- What rules must I follow when writing code? (Coding Conventions)
- What must I never do? (Do/Don't)
- What will go wrong if I'm not careful? (Common Pitfalls)

Claude does not browse README files during code generation unless explicitly asked. CLAUDE.md
is the single authoritative control file for agent behavior in this project.

### HANDOFF.md is written for humans

`memory/HANDOFF.md` is generated by the `handoff` skill at the end of a work session. It
captures where development left off, what decisions were made, and what the next engineer
should know before picking up the work. It is conversational, narrative, and often includes
context that would be noise in a machine-control file (e.g., "I tried approach X and it didn't
work because...").

This document is not read by Claude automatically. It is consulted by a human before starting
a new session, or by Claude when the human explicitly asks it to reload the handoff context.

## CLAUDE.md Structure Choices

### Optimized for AI reload

The document is structured so the most load-bearing sections come early: Project Overview,
then Build/Test/Lint Commands, then Validation. A new Claude session that only reads the first
half of CLAUDE.md gets enough to run tests and validate its work. The later sections (Common
Pitfalls, Security) are reference material consulted when something goes wrong.

Section headers use consistent imperative labels ("Do / Don't", "Validation") rather than
descriptive labels ("Notes", "Considerations") because imperative labels trigger more reliable
behavior in Claude.

### Prescriptive validation

The Validation section includes an explicit instruction: "Do NOT claim a task is done without
passing validation output." This is stronger than "run the tests before committing." Claude
will sometimes summarize what it expects tests to do rather than actually running them. The
explicit prohibition on claiming completion without evidence changes the default behavior.

### Behavioral Do/Don't

The Do/Don't section uses parallel imperative structure (Do: run validation, Don't: skip
guardrails) rather than prose paragraphs. This format is easier for Claude to pattern-match
against a proposed action. Prose instructions tend to be interpreted as suggestions; imperative
lists are interpreted as rules.

---

## HANDOFF.md Structure Choices

### Skill-generated, not hand-maintained

HANDOFF.md is generated by the `handoff` Claude Code skill rather than maintained manually.
This has two benefits. First, it is always current — generated at the end of the session that
produced it. Second, it inherits the structure defined in `claude_skills/handoff/SKILL.md`,
which means every handoff document has a consistent format: current state, decisions made,
open questions, next steps.

### Lives in memory/, not docs/

`memory/HANDOFF.md` lives in the `memory/` directory rather than alongside the design
documents in `docs/`. This reflects its different lifecycle: design documents are stable
references updated when the architecture changes; handoff documents are ephemeral session
artifacts that may be overwritten at the end of every session.

---

## Tradeoffs

### What is included vs. excluded from CLAUDE.md

CLAUDE.md includes everything Claude needs to write correct code and validate its work without
asking clarifying questions. It excludes:

- **Project history and rationale** — this belongs in design notes, not agent control files.
  Claude does not need to know why a decision was made; it needs to know what the decision is.
- **Tutorial-style explanations** — CLAUDE.md assumes Claude understands Python and the
  Anthropic SDK. Teaching concepts wastes space and dilutes the behavioral signal.
- **Changelog** — version history adds noise with no behavioral value for an agent.

### Depth vs. breadth

The Common Pitfalls section is deliberately specific rather than exhaustive. Six concrete
failure modes with remediation steps are more useful than twenty abstract warnings. The goal
is not completeness but actionability: each pitfall entry tells Claude (or a human) exactly
what to check and what to do differently.

The Folder Map is breadth-first rather than depth-first — it lists every directory and key
file without explaining the internals of each. This gives Claude a navigation map without
duplicating the inline documentation that already lives in the source files themselves.
